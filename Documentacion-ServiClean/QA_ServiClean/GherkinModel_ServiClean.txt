Requisito: RF-01

Escenario 1: Búsqueda exitosa con resultados disponibles.
  - Dado que el cliente se encuentra autenticado y en la vista frm_inicio_busqueda.
  - Cuando selecciona el filtro de fecha para "Mañana a las 08:00 AM" y presiona el botón "Buscar".
  - Entonces el sistema debe consultar la base de datos de perfiles.
  - Y mostrar una lista en pantalla con las colaboradoras que tengan su estado como "Disponible" en esa franja horaria.

Escenario 2: Búsqueda sin colaboradoras disponibles (Manejo de Empty State).
  - Dado que el cliente se encuentra en la vista frm_inicio_busqueda.
  - Cuando selecciona una fecha de alta demanda (ej. 24 de Diciembre a las 10:00 AM) donde la disponibilidad es cero.
  - Entonces el sistema debe mostrar un mensaje amigable indicando: "No hay colaboradoras disponibles en este horario".
  - Y sugerir al usuario el horario más cercano con disponibilidad.

Requisito: RF-09

Escenario 1: Retención preventiva de fondos (Autorización).
  - Dado que el cliente se encuentra en la pantalla frm_checkout_pago confirmando un servicio de $80,000 COP.
  - Cuando ingresa una tarjeta de crédito válida, con fondos suficientes, y hace clic en "Pagar y Reservar".

  - Y realizar una "retención" (hold) por los $80,000 COP sin debitarlos de la cuenta aún.
  - Y cambiar el estado de la reserva de "Pendiente" a "Confirmada".

Escenario 2: Liberación del pago (Captura).
  - Dado que existe una reserva en estado "Confirmada" con fondos retenidos.
  - Cuando el colaborador o el sistema marcan el servicio como "Finalizado" tras cumplir el horario.
  - Entonces el sistema debe notificar a la pasarela de pagos para hacer el cargo efectivo (captura) de los fondos retenidos.
  - Y generar el comprobante de pago para el cliente.

Requisito: RNF-01

Escenario 1: Almacenamiento seguro de contraseñas (Hashing).
  - Dado que un nuevo usuario o colaborador completa el formulario de registro.
  - Cuando el sistema procesa la solicitud para guardar la cuenta en la base de datos.
  - Entonces el servidor (backend) debe aplicar un algoritmo de hashing fuerte (ej. bcrypt) con un "salt" aleatorio.
  - Y almacenar únicamente el hash resultante en la base de datos, garantizando que ni siquiera los administradores del sistema puedan leer la contraseña original.

Escenario 2: Protección de datos financieros en tránsito y reposo.
  - Dado que un cliente ingresa los datos de su tarjeta de crédito para pagar una reserva.
  - Cuando la información viaja desde el navegador del cliente hasta el servidor y posteriormente se registra.
  - Entonces la transmisión debe realizarse forzosamente bajo el protocolo HTTPS (TLS 1.2 o superior).
  - Y cualquier registro en los logs del servidor o en la base de datos debe enmascarar el número de la tarjeta, dejando visibles únicamente los últimos 4 dígitos (ej. **** **** **** 4444).

Requisito: RNF-02

Escenario 1: Cumplimiento de latencia bajo concurrencia máxima esperada.
  - Dado que la plataforma ServiClean se encuentra operando en un entorno de producción o pre-producción (Staging).
  - Cuando 500 usuarios concurrentes ejecutan una petición al motor de búsqueda de empleadas al mismo tiempo.
  - Entonces la infraestructura del servidor (Balanceador de carga, Backend y Base de Datos) debe procesar las solicitudes de manera asíncrona.
  - Y garantizar que el tiempo máximo de respuesta para el usuario no supere los 3,000 milisegundos (3 segundos).

Escenario 2: Tolerancia a fallos sin denegación de servicio.
  - Dado que el sistema está siendo sometido a la prueba de los 500 usuarios concurrentes.
  - Cuando se analizan los registros (logs) de respuesta del servidor al finalizar la prueba.
  - Entonces la tasa de errores de conexión (timeouts o caídas de base de datos) debe ser del 0%.
  - Y todos los usuarios deben recibir una respuesta HTTP 200 (OK) con el JSON de los resultados de su búsqueda.
